Testing y Performance en Share Notes

Garantizando calidad con Jest, Supertest y buenas prácticas de rendimiento

🧪 1. Testing Backend con Jest + Supertest

Los tests se ejecutan contra una base de datos de prueba real (MongoDB). No usamos mocks de base de datos, sino una BD dedicada.

Configuración del Proyecto

En package.json, Jest está configurado así:

// package.json (jest config)
"jest": {
  "testEnvironment": "node",
  "testMatch": ["**/tests/**/*.test.js"],
  "setupFiles": ["dotenv/config"],
  "transform": {}
}

// Script de test
"test": "cross-env NODE_OPTIONS=--experimental-vm-modules jest --detectOpenHandles --forceExit"
¿Por qué --experimental-vm-modules? El proyecto usa ES Modules ("type": "module" en package.json). Jest necesita esta bandera para importar con import/export.
Supertest

Pruebas de Integración

Usamos Supertest para simular peticiones HTTP reales contra la app Express sin necesidad de levantar el servidor.

import request from "supertest";
import app from "../app.js";

const API = "/api";

// beforeAll: Conectar a MongoDB de prueba
beforeAll(async () => {
  await mongoose.connect(process.env.MONGO_URI_TEST);
});

// afterEach: Limpiar datos entre tests
afterEach(async () => {
  await User.deleteMany({});
  await Category.deleteMany({});
});
Autenticación

Manejo de Cookies en Tests

La API usa cookies httpOnly para autenticación. Los tests deben capturar y reenviar la cookie manualmente.

const loginAndGetCookie = async () => {
  // Registrar usuario
  await request(app).post(`${API}/auth/register`).send({
    name: "Test User",
    email: "test@test.com",
    password: "password123",
  });

  // Login y capturar cookie
  const res = await request(app)
    .post(`${API}/auth/login`)
    .send({ email: "test@test.com", password: "password123" });

  const cookie = res.headers["set-cookie"];
  return { cookie, userId: res.body.user._id };
};

📋 2. Estructura de Tests: Auth

Los tests de autenticación cubren los 3 estados principales de cada endpoint:

✅ Register - Happy Path

Registro exitoso devuelve 201 con datos del usuario. No debe devolver cookie ni token en el body.

it("Debe registrar y devolver 201 sin cookie", async () => {
  const res = await request(app).post(`${API}/auth/register`).send({
    name: "Ana García",
    email: "ana@test.com",
    password: "secret123",
  });

  expect(res.status).toBe(201);
  expect(res.body.user).not.toHaveProperty("password");
  expect(res.headers["set-cookie"]).toBeUndefined();
});
❌ Register - Error Path

Email duplicado devuelve 400 con mensaje descriptivo.

it("Debe retornar 400 si el email ya existe", async () => {
  await registerUser();
  const res = await request(app).post(`${API}/auth/register`).send({
    name: "Otro",
    email: "test@test.com",
    password: "pass456",
  });

  expect(res.status).toBe(400);
  expect(res.body.message).toMatch(/registrado/i);
});
❌ Login - Credenciales Inválidas

Contraseña incorrecta devuelve 401.

it("Debe retornar 401 con contraseña incorrecta", async () => {
  await request(app).post(`${API}/auth/register`).send({
    name: "Test",
    email: "wrong@test.com",
    password: "correctpass",
  });

  const res = await request(app).post(`${API}/auth/login`).send({
    email: "wrong@test.com",
    password: "wrongpass",
  });

  expect(res.status).toBe(401);
  expect(res.body.message).toMatch(/credenciales/i);
});
✅ Login - Happy Path

Login exitoso devuelve 200 con cookie Set-Cookie.

it("Debe iniciar sesión y devolver 200 con cookie", async () => {
  await registerUser();

  const res = await request(app).post(`${API}/auth/login`).send({
    email: "test@test.com",
    password: "password123",
  });

  expect(res.status).toBe(200);
  expect(res.body).toHaveProperty("user");
  expect(res.headers["set-cookie"]).toBeDefined();
});

📝 3. Tests de Notas

Flujo Completo

Los tests de notas verifican desde la autenticación hasta la creación con categoría:

// Test de autorización
it("401 sin autenticación", async () => {
  const res = await request(app).get(`${API}/notes`);
  expect(res.status).toBe(401);
});

// Test de lista vacía
it("Lista vacía para nuevo usuario", async () => {
  const { cookie } = await loginAndGetCookie();
  const res = await request(app).get(`${API}/notes`).set("Cookie", cookie);
  expect(res.body.notes.length).toBe(0);
});

// Test de creación con categoría
it("Crear nota y retornar 201", async () => {
  const { cookie, userId } = await loginAndGetCookie();
  const category = await Category.create({
    name: "Matemáticas", color: "#6366f1", user: userId,
  });

  const res = await request(app)
    .post(`${API}/notes`)
    .set("Cookie", cookie)
    .send({ title: "Álgebra", content: "Matrices...", category: category._id });

  expect(res.status).toBe(201);
  expect(res.body.note.title).toBe("Álgebra");
});

BD de Prueba

Usamos MONGO_URI_TEST del .env para aislar los tests de la base de datos de desarrollo.

// .env
MONGO_URI_TEST=mongodb://127.0.0.1:27017/share_notes_test

// En los tests:
beforeAll(async () => {
  await mongoose.connect(process.env.MONGO_URI_TEST);
});

afterAll(async () => {
  await mongoose.connection.dropDatabase();
  await mongoose.connection.close();
});
Importante: El dropDatabase() en afterAll limpia todo después de los tests. Nunca afecta a la BD de desarrollo.

🚀 4. Rendimiento: Código que Vuela

Static Analysis

Buenas Prácticas

  • Índices en MongoDB: Las colecciones tienen índices en campos de búsqueda frecuente (tags, reminder.date, texto completo)
  • Select +password: La contraseña solo se trae cuando es necesario (.select("+password"))
  • Soft Delete: Las notas no se eliminan realmente, solo se marca deletedAt
Optimización

API Performance

  • Paginación: Las listas de notas están paginadas
  • Population selectiva: Solo se traen los campos necesarios de referencias
  • Compresión: Respuestas JSON ligeras sin datos sensibles
Frontend

React Performance

  • Zustand: Estado global ligero sin re-renders innecesarios
  • Vite: Build con Tree Shaking elimina código muerto
  • Lazy loading: Las rutas del dashboard cargan bajo demanda

🚩 Guía de Errores: "Profe, no funciona"

01. jwt malformed

Causa: La cookie token está corrupta o no se envió correctamente.

02. Email ya registrado (400)

Causa: El email ya existe en la BD. Cada email debe ser único.

03. Error al enviar email (SMTP)

Causa: Credenciales SMTP incorrectas o Contraseña de Aplicación inválida.

04. require is not defined

Causa: Usar require en un proyecto ESM ("type": "module"). Usa import/export.

05. Token inválido o expirado

Causa: El token de reset de contraseña expiró (1 hora) o ya fue usado.

06. CORS Policy Error

Causa: El frontend y backend están en distintos puertos/orígenes y CORS no está configurado.

07. Cookies no se guardan

Causa: Falta withCredentials: true en Axios o sameSite: 'none' en la cookie.

08. MongoDB Connection Error

Causa: MongoDB Compass no está abierto o la URI es incorrecta.

09. 429 Too Many Requests

Causa: Rate limit activado. En Render, agregar app.set("trust proxy", 1).

10. Mocha/Chai no encontrado

Causa: El proyecto no usa Mocha, usa Jest + Supertest. Ejecuta npm test.

11. Module not found (~/...)

Causa: Jest no entiende los alias. Revisa moduleNameMapper en la config de Jest.

12. SIGTERM (Render)

Causa: La app consumió mucha memoria. Optimiza el build o cambia de plan.
